# UC20-T3: Circuit Breaking

## Objective

Demonstrate that a `DestinationRule` with restrictive connection pool limits, enforced by the **waypoint proxy**, triggers circuit breaking (HTTP 503 with `UO` flag) when concurrent traffic exceeds the configured thresholds.

## Prerequisites

- Both clusters running with bookinfo deployed and accessible
- `generate-traffic.sh` running for Kiali visualization (recommended)
- Kiali open (OSSMC via ACM console):
  https://console-openshift-console.apps.cluster-72nh2.dynamic.redhatworkshops.io/ossmconsole/graph

## Quick Run

```bash
./ossm/uc20-t3-verify.sh
```

## Manual Steps

### 1. Verify baseline — traffic flows normally

```bash
curl -s -o /dev/null -w "EAST: HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200.

### 2. Deploy reviews-waypoint (L7 proxy)

DestinationRule circuit breaking is enforced by the waypoint proxy. Deploy it on demand:

```bash
oc --context east apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: reviews-waypoint
  namespace: bookinfo
  labels:
    istio.io/waypoint-for: service
spec:
  gatewayClassName: istio-waypoint
  listeners:
    - name: mesh
      port: 15008
      protocol: HBONE
EOF

oc --context east label svc reviews -n bookinfo istio.io/use-waypoint=reviews-waypoint --overwrite
oc --context east wait --for=condition=Ready pod -l gateway.networking.k8s.io/gateway-name=reviews-waypoint -n bookinfo --timeout=60s
```

### 3. Apply DestinationRule with restrictive limits

```bash
oc --context east apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: reviews-circuit-breaker
  namespace: bookinfo
spec:
  host: reviews.bookinfo.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 1
      http:
        h2UpgradePolicy: DO_NOT_UPGRADE
        http1MaxPendingRequests: 1
        http2MaxRequests: 1
        maxRequestsPerConnection: 1
    outlierDetection:
      consecutive5xxErrors: 1
      interval: 5s
      baseEjectionTime: 30s
      maxEjectionPercent: 100
EOF
```

### 4. Verify DestinationRule propagated to waypoint

```bash
REVIEWS_WP=$(oc --context east get pods -n bookinfo -l gateway.networking.k8s.io/gateway-name=reviews-waypoint -o jsonpath='{.items[0].metadata.name}')
oc --context east exec -n bookinfo "$REVIEWS_WP" -- pilot-agent request GET "config_dump" 2>/dev/null \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
for c in data.get('configs', []):
    for cluster in c.get('dynamic_active_clusters', []):
        cl = cluster.get('cluster', {})
        if 'reviews' in cl.get('name', '') and 'waypoint' not in cl.get('name', ''):
            cb = cl.get('circuit_breakers', {})
            print(f'Cluster: {cl[\"name\"]}')
            print(json.dumps(cb, indent=2))
"
```

Expected: `max_connections: 1`, `max_pending_requests: 1`, `max_requests: 1`.

### 5. Generate concurrent traffic to trigger circuit breaker

From inside the productpage pod, send 20 simultaneous requests to reviews:

```bash
PRODUCTPAGE_POD=$(oc --context east get pods -n bookinfo -l app=productpage -o jsonpath='{.items[0].metadata.name}')
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request, concurrent.futures, json

def call_reviews(i):
    try:
        req = urllib.request.Request('http://reviews:9080/reviews/0')
        with urllib.request.urlopen(req, timeout=10) as resp:
            return (i, resp.status)
    except urllib.error.HTTPError as e:
        return (i, e.code)
    except Exception as e:
        return (i, str(e))

with concurrent.futures.ThreadPoolExecutor(max_workers=20) as pool:
    futures = [pool.submit(call_reviews, i) for i in range(20)]
    results = [f.result() for f in concurrent.futures.as_completed(futures)]

status_count = {}
for idx, status in results:
    status_count[status] = status_count.get(status, 0) + 1

print(json.dumps(status_count, indent=2))
"
```

Expected: majority of requests return **503** (circuit breaker open).

### 6. Confirm circuit breaker via waypoint metrics

```bash
oc --context east exec -n bookinfo "$REVIEWS_WP" -- pilot-agent request GET "stats?filter=response_flags" 2>/dev/null \
  | grep "UO"
```

Expected: `response_flags.UO` counter > 0 (UO = Upstream Overflow = circuit breaker triggered).

### 7. Cleanup

```bash
oc --context east delete destinationrule reviews-circuit-breaker -n bookinfo
oc --context east label svc reviews -n bookinfo istio.io/use-waypoint-
oc --context east delete gateway reviews-waypoint -n bookinfo
```

### 8. Verify recovery

```bash
curl -s -o /dev/null -w "EAST: HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200 — normal operation restored.

## Expected Results

| Phase | Result |
|-------|--------|
| Baseline | HTTP 200 |
| DestinationRule applied | `max_connections: 1` visible in waypoint Envoy config |
| 20 concurrent requests | Majority return **503** (circuit breaker) |
| Waypoint metrics | `response_flags.UO` count matches 503s |
| After cleanup | HTTP 200 — instant recovery |

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| DestinationRule | Defines connection pool limits and outlier detection | Yes — traffic policy |
| reviews-waypoint | Envoy proxy that enforces circuit breaker thresholds | Yes — L7 data plane |
| ztunnel | Routes traffic through waypoint (HBONE) | Yes — L4 data plane |
| istiod | Pushes DestinationRule config to waypoint's Envoy | Yes — control plane |
| Circuit breaker (UO flag) | Envoy rejects excess requests with 503 | Yes — resilience |
| Kiali | Shows error rate spike during circuit breaking | Yes — mesh observability |

## Key Takeaway

In ambient mode, circuit breaking is enforced by the **waypoint proxy** (not ztunnel). A `DestinationRule` with restrictive connection pool limits causes the waypoint's Envoy to reject excess concurrent requests with HTTP 503 and the `UO` (Upstream Overflow) response flag. This provides resilience without any application changes — the mesh protects downstream services from being overwhelmed.
