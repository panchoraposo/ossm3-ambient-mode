# UC20-T2: Request Timeouts

## Objective

Demonstrate that the **waypoint proxy** enforces request timeouts configured via `HTTPRoute` (Gateway API). When a downstream dependency (ratings) is slow, the waypoint cuts the response at the configured timeout, returning HTTP 504 (`upstream request timeout`) instead of waiting indefinitely.

## Prerequisites

- Both clusters running with bookinfo deployed and accessible
- Waypoint proxies deployed and processing traffic (reviews-waypoint, ratings-waypoint)
- `generate-traffic.sh` running for Kiali visualization (recommended — shows 504 errors on reviews-v2/v3)
- Kiali open (OSSMC via ACM console):
  https://console-openshift-console.apps.cluster-72nh2.dynamic.redhatworkshops.io/ossmconsole/graph

## Quick Run

```bash
./ossm/uc20-t2-verify.sh
```

## Architecture

The bookinfo topology makes this test interesting:

```
productpage → reviews-v1 → (no ratings call)         → fast response
productpage → reviews-v2 → ratings (delayed 5s)      → slow → timeout!
productpage → reviews-v3 → ratings (delayed 5s)      → slow → timeout!
```

- `reviews-v1` does NOT call ratings → responds instantly, unaffected by the delay
- `reviews-v2` and `v3` call ratings → blocked by the 5s delay → timeout triggers at 2s

This demonstrates how timeouts protect upstream services from cascading slowness in downstream dependencies.

### APIs used

| Resource | API | Purpose |
|----------|-----|---------|
| HTTPRoute (`reviews-timeout`) | Gateway API (GA) | `timeouts.request: 2s` on reviews at the waypoint |
| VirtualService (`ratings-delay`) | Istio (TP) | Fault injection: 5s delay on ratings to simulate slowness |

The timeout uses HTTPRoute (GA, standard Kubernetes API). The fault injection to create the slow dependency uses VirtualService because fault injection is not available in Gateway API.

## Manual Steps

### 1. Verify baseline — all reviews versions respond fast

```bash
PRODUCTPAGE_POD=$(oc --context east get pods -n bookinfo -l app=productpage -o jsonpath='{.items[0].metadata.name}')
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request, time
for i in range(5):
    start = time.time()
    req = urllib.request.Request('http://reviews:9080/reviews/0')
    try:
        with urllib.request.urlopen(req, timeout=10) as resp:
            print(f'  [{i+1}] Status: {resp.status}, Elapsed: {time.time()-start:.2f}s')
    except urllib.error.HTTPError as e:
        print(f'  [{i+1}] Status: {e.code}, Elapsed: {time.time()-start:.2f}s')
"
```

Expected: all responses fast (< 1s), mixed 200/404.

### 2. Inject delay on ratings (make reviews-v2/v3 slow)

```bash
oc --context east apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: ratings-delay
  namespace: bookinfo
spec:
  hosts:
  - ratings.bookinfo.svc.cluster.local
  http:
  - fault:
      delay:
        fixedDelay: 5s
        percentage:
          value: 100
    route:
    - destination:
        host: ratings.bookinfo.svc.cluster.local
EOF
```

### 3. Set timeout on reviews via HTTPRoute (2 seconds)

```bash
oc --context east apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: reviews-timeout
  namespace: bookinfo
spec:
  parentRefs:
  - kind: Service
    group: ""
    name: reviews
    port: 9080
  rules:
  - timeouts:
      request: 2s
    backendRefs:
    - name: reviews
      port: 9080
EOF
```

### 4. Test — reviews-v2/v3 get 504, reviews-v1 works fine

```bash
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request, time
for i in range(6):
    start = time.time()
    req = urllib.request.Request('http://reviews:9080/reviews/0')
    try:
        with urllib.request.urlopen(req, timeout=10) as resp:
            elapsed = time.time() - start
            print(f'  [{i+1}] Status: {resp.status}, Elapsed: {elapsed:.2f}s')
    except urllib.error.HTTPError as e:
        elapsed = time.time() - start
        body = e.read().decode()[:40]
        print(f'  [{i+1}] Status: {e.code}, Elapsed: {elapsed:.2f}s — {body}')
"
```

Expected:
- Some responses: **HTTP 504** at **~2s** (`upstream request timeout`) — these hit reviews-v2/v3
- Some responses: **HTTP 200** at **< 0.1s** — these hit reviews-v1 (no ratings call)

### 5. Cleanup

```bash
oc --context east delete httproute reviews-timeout -n bookinfo
oc --context east delete virtualservice ratings-delay -n bookinfo
```

### 6. Verify recovery

```bash
curl -s -o /dev/null -w "EAST: HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200, normal latency.

## Expected Results

| Phase | reviews-v1 | reviews-v2/v3 |
|-------|-----------|---------------|
| Baseline | 200, fast | 200, fast |
| Delay on ratings + HTTPRoute timeout 2s | 200, fast (no ratings) | **504, ~2s** (timeout) |
| After cleanup | 200, fast | 200, fast |

> **Alternativa VirtualService**: El timeout también puede configurarse con `VirtualService` (`http[].timeout: 2s`). El resultado es idéntico. VirtualService es Technology Preview en OSSM 3.2 ambient mode.

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| HTTPRoute (timeouts) | Configures max request duration on reviews (2s) | Yes — traffic management (GA) |
| VirtualService (fault delay) | Simulates slow ratings service (5s) | Yes — resilience testing (TP) |
| reviews-waypoint | Enforces timeout, returns 504 when exceeded | Yes — L7 data plane |
| ratings-waypoint | Injects delay before forwarding to ratings | Yes — L7 data plane |
| ztunnel | Routes traffic through waypoints (HBONE) | Yes — L4 data plane |
| istiod | Pushes HTTPRoute/VirtualService config to waypoints | Yes — control plane |

## Key Takeaway

Request timeouts in ambient mode are enforced by the **waypoint proxy** via `HTTPRoute` (Gateway API). When a downstream service is slow, the waypoint cuts the request at the configured timeout and returns HTTP 504. This prevents cascading latency — the caller gets a fast failure instead of waiting indefinitely. The timeout is transparent to the application and applies selectively based on the actual dependency chain (reviews-v1 is unaffected because it doesn't call the slow service). Using HTTPRoute (GA, standard Kubernetes API) for timeouts means this feature is fully supported and does not depend on Istio-specific CRDs.
