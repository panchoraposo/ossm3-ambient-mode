# UC13: Local-First Traffic Awareness (API Routing Without External LB/DNS)

> **Alcance**: El caso de uso original plantea contexto multi-cluster (EAST→WEST). En OSSM 3.2 (Istio 1.27) ambient mode, el data plane cross-cluster no soporta waypoints, por lo que aquí se demuestra la misma funcionalidad single-cluster (Step 1 del PDF: verificar que el tráfico intra-cluster se mantiene local sin LB/DNS externo). Los Steps 2-3 (cross-cluster migration y RHCL) se extenderán con Istio 1.29+.

## Objective

Demonstrate that when services communicate within the same cluster in ambient mode, traffic stays entirely local — routed by ztunnel (L4) without hitting any external load balancer, DNS server, or east-west gateway. Prove this through ztunnel access logs and DNS resolution evidence.

## Architecture

```
  productpage pod                          reviews pod
       │                                       ▲
       ▼                                       │
   ztunnel ──── mTLS (HBONE) ──────────── ztunnel
   (node A)       within cluster          (node B)
                                            
   ✔ Internal CoreDNS (cluster.local)
   ✘ No external DNS
   ✘ No F5 / external LB
   ✘ No East-West Gateway
```

- Traffic from `productpage` to `reviews.bookinfo.svc.cluster.local` is intercepted by ztunnel
- ztunnel resolves the destination to a local pod IP within the cluster CIDR
- mTLS (HBONE) is established directly between the source and destination ztunnels
- No east-west gateway is involved when the service exists locally

## Test Flow

### Step 1: Enable ztunnel access logging

Apply a Telemetry resource to capture access logs from ztunnel:

```bash
oc --context east apply -f - <<EOF
apiVersion: telemetry.istio.io/v1
kind: Telemetry
metadata:
  name: ztunnel-logging
  namespace: istio-system
spec:
  selector:
    matchLabels:
      app: ztunnel
  accessLogging:
    - providers:
        - name: envoy
      filter:
        expression: "true"
EOF
```

### Step 2: Generate intra-cluster traffic

Send requests from productpage to reviews within the EAST cluster:

```bash
curl http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

### Step 3: Verify DNS resolution is internal

From a bookinfo pod, resolve `reviews.bookinfo.svc.cluster.local` and confirm it resolves via CoreDNS (ClusterIP `172.30.0.10`) — not via external DNS:

```bash
oc --context east exec -n bookinfo deploy/productpage-v1 -- \
  cat /etc/resolv.conf
# → nameserver 172.30.0.10 (cluster CoreDNS)

oc --context east exec -n bookinfo deploy/productpage-v1 -- \
  python3 -c "import socket; print(socket.gethostbyname('reviews.bookinfo.svc.cluster.local'))"
# → ClusterIP within 172.30.x.x range
```

### Step 4: Verify ztunnel logs — traffic stayed local

Check ztunnel access logs for reviews traffic. Source and destination IPs should both be within the cluster Pod CIDR (`10.128.x.x` / `10.129.x.x`):

```bash
oc --context east logs -n ztunnel ds/ztunnel --tail=50 2>/dev/null | grep "reviews"
```

Expected log pattern:
- `src.addr` = pod IP of productpage (e.g., `10.128.x.x`)
- `dst.addr` = pod IP of reviews (e.g., `10.129.x.x`)
- No mention of east-west gateway IPs or external addresses

### Step 5: Verify no east-west gateway traffic

Confirm the east-west gateway had zero traffic related to reviews:

```bash
oc --context east logs -n istio-system deploy/istio-eastwestgateway --tail=50 2>/dev/null | grep "reviews"
# → empty (no cross-cluster routing)
```

### Step 6: Cleanup

```bash
oc --context east delete telemetry ztunnel-logging -n istio-system
```

## Expected Results

| Verification | Expected result |
|-------------|----------------|
| DNS nameserver | `172.30.0.10` (CoreDNS, cluster-internal) |
| DNS resolution of reviews | ClusterIP `172.30.x.x` (internal) |
| ztunnel src.addr | Pod CIDR `10.128.x.x` or `10.129.x.x` |
| ztunnel dst.addr | Pod CIDR `10.128.x.x` or `10.129.x.x` |
| East-West gateway logs for reviews | Empty — no cross-cluster traffic |
| Pod restarts | 0 |

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| ztunnel | Intercepts traffic, establishes mTLS, routes locally | Yes — L4 data plane |
| istiod | Service registry, endpoint resolution | Yes — control plane |
| HBONE (mTLS) | Encrypted tunnel between ztunnels | Yes — ambient security |
| CoreDNS | Standard Kubernetes DNS | Kubernetes (support) |

The key takeaway is that ztunnel + istiod route traffic to local endpoints automatically. When the service exists in the same cluster, traffic never leaves the cluster boundary — no external DNS, no F5, no east-west gateway.

## Key Takeaway

In ambient mode, intra-cluster communication is handled entirely by ztunnel (L4 mTLS). Service discovery uses standard Kubernetes CoreDNS and istiod's endpoint registry. Traffic is encrypted, authenticated, and routed locally without involving any external infrastructure. This proves that for local APIs, the mesh provides zero-dependency routing — no external load balancers, no external DNS queries, no additional network hops.
