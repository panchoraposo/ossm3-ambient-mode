# UC20-T1: Fault Injection (Delay & Abort)

## Objective

Demonstrate that the **waypoint proxy** can inject faults (HTTP aborts and artificial delays) into service-to-service traffic using `VirtualService`. This allows testing application resilience without modifying code.

## Prerequisites

- Both clusters running with bookinfo deployed and accessible
- `generate-traffic.sh` running for Kiali visualization (recommended — shows increased latency and error rate)
- Kiali open (OSSMC via ACM console):
  https://console-openshift-console.apps.cluster-72nh2.dynamic.redhatworkshops.io/ossmconsole/graph

> **Note**: `VirtualService` is Technology Preview (TP) in OSSM 3.2 ambient mode. This test demonstrates it works with waypoint proxies.

## Quick Run

```bash
./ossm/uc20-t1-verify.sh
```

## Manual Steps

### Deploy reviews-waypoint (L7 proxy)

VirtualService fault injection requires a waypoint proxy. Deploy it on demand:

```bash
oc --context east apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: reviews-waypoint
  namespace: bookinfo
  labels:
    istio.io/waypoint-for: service
spec:
  gatewayClassName: istio-waypoint
  listeners:
    - name: mesh
      port: 15008
      protocol: HBONE
EOF

oc --context east label svc reviews -n bookinfo istio.io/use-waypoint=reviews-waypoint --overwrite
oc --context east wait --for=condition=Ready pod -l gateway.networking.k8s.io/gateway-name=reviews-waypoint -n bookinfo --timeout=60s
```

### Phase A: Fault Abort (HTTP 500)

#### 1. Verify baseline

```bash
PRODUCTPAGE_POD=$(oc --context east get pods -n bookinfo -l app=productpage -o jsonpath='{.items[0].metadata.name}')
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request
req = urllib.request.Request('http://reviews:9080/reviews/0')
with urllib.request.urlopen(req, timeout=10) as resp:
    print(f'Status: {resp.status}')
"
```

Expected: `Status: 200`.

#### 2. Apply VirtualService with abort fault

Inject HTTP 500 on 100% of requests to reviews:

```bash
oc --context east apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-fault-inject
  namespace: bookinfo
spec:
  hosts:
  - reviews.bookinfo.svc.cluster.local
  http:
  - fault:
      abort:
        httpStatus: 500
        percentage:
          value: 100
    route:
    - destination:
        host: reviews.bookinfo.svc.cluster.local
EOF
```

#### 3. Verify abort — reviews returns 500

```bash
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request
try:
    req = urllib.request.Request('http://reviews:9080/reviews/0')
    with urllib.request.urlopen(req, timeout=10) as resp:
        print(f'Status: {resp.status}')
except urllib.error.HTTPError as e:
    print(f'Status: {e.code}')
    print(f'Body: {e.read().decode()[:100]}')
"
```

Expected: `Status: 500`, `Body: fault filter abort`.

#### 4. Cleanup abort

```bash
oc --context east delete virtualservice reviews-fault-inject -n bookinfo
```

### Phase B: Fault Delay (5 seconds)

#### 5. Apply VirtualService with delay fault

Inject 5-second delay on 100% of requests to reviews:

```bash
oc --context east apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: reviews-fault-inject
  namespace: bookinfo
spec:
  hosts:
  - reviews.bookinfo.svc.cluster.local
  http:
  - fault:
      delay:
        fixedDelay: 5s
        percentage:
          value: 100
    route:
    - destination:
        host: reviews.bookinfo.svc.cluster.local
EOF
```

#### 6. Verify delay — response takes ~5 seconds

```bash
oc --context east exec -n bookinfo "$PRODUCTPAGE_POD" -- python3 -c "
import urllib.request, time
start = time.time()
req = urllib.request.Request('http://reviews:9080/reviews/0')
with urllib.request.urlopen(req, timeout=15) as resp:
    elapsed = time.time() - start
    print(f'Status: {resp.status}')
    print(f'Elapsed: {elapsed:.2f}s')
"
```

Expected: `Status: 200`, `Elapsed: ~5.0s`.

#### 7. Cleanup delay

```bash
oc --context east delete virtualservice reviews-fault-inject -n bookinfo
```

### 8. Cleanup waypoint and verify recovery

```bash
oc --context east label svc reviews -n bookinfo istio.io/use-waypoint-
oc --context east delete gateway reviews-waypoint -n bookinfo
curl -s -o /dev/null -w "EAST: HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200 — normal latency restored, L4-only baseline.

## Expected Results

| Phase | Request | Result |
|-------|---------|--------|
| Baseline | productpage → reviews | HTTP 200, fast |
| Abort fault (100%) | productpage → reviews | **HTTP 500** (`fault filter abort`) |
| Delay fault (5s, 100%) | productpage → reviews | HTTP 200, **~5s delay** |
| After cleanup | productpage → reviews | HTTP 200, fast |

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| VirtualService | Defines fault injection rules (abort/delay) | Yes — traffic management (TP) |
| reviews-waypoint | Envoy proxy that injects faults before reaching reviews | Yes — L7 data plane |
| ztunnel | Routes traffic through waypoint (HBONE) | Yes — L4 data plane |
| istiod | Pushes VirtualService config to waypoint's Envoy | Yes — control plane |
| Fault filter | Envoy HTTP filter that aborts or delays requests | Yes — resilience testing |
| Kiali | Shows error rate spikes and latency increases | Yes — mesh observability |

## Key Takeaway

Fault injection in ambient mode is enforced by the **waypoint proxy** via `VirtualService`. The waypoint's Envoy applies fault filters (abort with configurable HTTP status, or delay with configurable duration) before forwarding requests to the destination. This enables chaos engineering and resilience testing without modifying application code — the mesh injects faults transparently at the network layer.
