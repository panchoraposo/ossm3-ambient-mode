# UC20-T4: Request Authentication (JWT)

## Objective

Demonstrate that the Istio ingress gateway enforces JWT (JSON Web Token) validation on incoming requests using `RequestAuthentication` and `AuthorizationPolicy`. Requests without a valid JWT are rejected with HTTP 403, requests with an invalid JWT get HTTP 401, and requests with a valid token pass through normally.

## Prerequisites

- Both clusters running with bookinfo deployed and accessible
- `generate-traffic.sh` running for Kiali visualization (recommended — shows traffic flow when policies are applied)
- Kiali open (OSSMC via ACM console):
  https://console-openshift-console.apps.cluster-72nh2.dynamic.redhatworkshops.io/ossmconsole/graph

## Quick Run

```bash
./ossm/uc20-t4-verify.sh
```

## Architecture Note

In the current bookinfo topology, external traffic arrives via the **bookinfo-gateway** (Istio Gateway, class `istio`). This is a standalone Envoy pod that receives traffic from the OpenShift Router and forwards it to backend services. JWT validation policies target this gateway using `targetRefs` with `kind: Gateway`.

The traffic flow is:

```
External client → OpenShift Router → bookinfo-gateway (Envoy) → productpage
                                     ↑ JWT validated here
```

## Manual Steps

### 1. Verify baseline — unauthenticated traffic flows

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200 (no JWT required yet).

### 2. Apply RequestAuthentication to the ingress gateway

This tells the gateway to validate JWTs from the Istio test issuer. Requests without a token are still accepted (RequestAuthentication only validates tokens when present):

```bash
oc --context east apply -f - <<EOF
apiVersion: security.istio.io/v1
kind: RequestAuthentication
metadata:
  name: bookinfo-gateway-jwt
  namespace: bookinfo
spec:
  targetRefs:
  - kind: Gateway
    group: gateway.networking.k8s.io
    name: bookinfo-gateway
  jwtRules:
  - issuer: "testing@secure.istio.io"
    jwksUri: "https://raw.githubusercontent.com/istio/istio/release-1.27/security/tools/jwt/samples/jwks.json"
EOF
```

### 3. Apply AuthorizationPolicy to require JWT

This enforces that only requests with a valid JWT (`requestPrincipals: ["*"]`) are allowed:

```bash
oc --context east apply -f - <<EOF
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: bookinfo-gateway-require-jwt
  namespace: bookinfo
spec:
  targetRefs:
  - kind: Gateway
    group: gateway.networking.k8s.io
    name: bookinfo-gateway
  rules:
  - from:
    - source:
        requestPrincipals: ["*"]
EOF
```

### 4. Test — request WITHOUT JWT is rejected

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: **HTTP 403** (Forbidden — no JWT present).

### 5. Get a valid test JWT

```bash
TOKEN=$(curl -s https://raw.githubusercontent.com/istio/istio/release-1.27/security/tools/jwt/samples/demo.jwt)
echo "$TOKEN" | cut -d'.' -f2 | base64 -d 2>/dev/null | python3 -m json.tool
```

This shows the token claims (issuer: `testing@secure.istio.io`).

### 6. Test — request WITH valid JWT passes

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" \
  -H "Authorization: Bearer $TOKEN" \
  http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: **HTTP 200** (valid JWT, request allowed).

### 7. Test — request with INVALID JWT is rejected

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" \
  -H "Authorization: Bearer invalid.token.here" \
  http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: **HTTP 401** (Unauthorized — invalid token signature).

### 8. Cleanup

```bash
oc --context east delete requestauthentication bookinfo-gateway-jwt -n bookinfo
oc --context east delete authorizationpolicy bookinfo-gateway-require-jwt -n bookinfo
```

### 9. Verify recovery

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/productpage
```

Expected: HTTP 200 — back to normal (no JWT required).

## Expected Results

| Phase | Request | Result |
|-------|---------|--------|
| Baseline (no policy) | No JWT | HTTP 200 |
| After JWT policy | No JWT | **HTTP 403** (Forbidden) |
| After JWT policy | Valid JWT | **HTTP 200** |
| After JWT policy | Invalid JWT | **HTTP 401** (Unauthorized) |
| After cleanup | No JWT | HTTP 200 |

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| RequestAuthentication | Defines JWT issuer and JWKS validation rules | Yes — mesh security |
| AuthorizationPolicy | Requires valid requestPrincipal (from JWT) | Yes — mesh security |
| bookinfo-gateway (Envoy) | Ingress gateway that validates JWT on incoming requests | Yes — L7 ingress |
| istiod | Pushes auth policies to gateway's Envoy config | Yes — control plane |
| JWKS endpoint | Public key set for JWT signature verification | External dependency |
| Kiali | Shows request errors when JWT is missing/invalid | Yes — mesh observability |

## Key Takeaway

JWT authentication in the service mesh is enforced at the **ingress gateway** level. The `RequestAuthentication` resource defines trusted JWT issuers, while the `AuthorizationPolicy` with `requestPrincipals: ["*"]` makes the JWT mandatory. This provides zero-trust request-level authentication at the mesh perimeter — no application code changes needed. Invalid tokens get HTTP 401, missing tokens get HTTP 403, and only valid tokens pass through to the application.
