# UC11: Special Ciphers (Legacy TLS Origination) ‚Äî Ambient (OSSM 3.2)

## Objective

Demonstrate how **OSSM 3.2 (Istio Ambient Mode)** can keep the mesh **modern and secure** while still enabling connectivity to an external/legacy backend that requires **TLS 1.0 + legacy cipher suites**, by **isolating** legacy behavior to the egress path.

Also replicate a ‚Äú3scale-like‚Äù pattern where a **special header** enables a ‚Äúcompat‚Äù route **only** for that destination (without lowering the overall mesh security posture).

## What this use case shows

- A ‚Äúlegacy‚Äù backend that **only accepts TLS 1.0** and a legacy cipher (default: `ECDHE-RSA-AES256-SHA`)
- An **egress waypoint** (ambient) that centralizes egress and performs L7 routing (header-based)
- **Header-based downgrade**:
  - no header ‚Üí ‚Äúmodern/strict‚Äù path ‚Üí fails
  - with header ‚Üí ‚Äúcompat‚Äù path ‚Üí works
- Server-side validation: legacy TLS server logs show **TLS 1.0 Handshake** and the negotiated cipher

## Architecture (high level)

```
workload (mesh) ‚îÄ‚îÄHTTP‚îÄ‚îÄ> ztunnel ‚îÄ‚îÄHBONE/mTLS‚îÄ‚îÄ> egress waypoint (Envoy)
                                         ‚îÇ
                                         ‚îî‚îÄ L7 header-based routing
                                              ‚îú‚îÄ strict connector  ‚Üí FAIL (forces TLS >=1.2 to backend)
                                              ‚îî‚îÄ compat connector  ‚Üí OK   (originates TLS1.0 + legacy cipher)
```

**Important**: the workload sends **HTTP** to the external host registered via `ServiceEntry`. Legacy TLS origination is implemented using dedicated ‚Äúconnectors‚Äù (HAProxy) to avoid relying on fields that are not exposed/available via `DestinationRule` in OSSM 3.2.

## Connectivity Link (Kuadrant) value-add

Builds on top of the Istio-only scenario and demonstrates added value:

- In pure Istio, the ‚Äúdowngrade‚Äù is controlled by the client (header).
- With Connectivity Link, you can **control who** can access the ‚Äúlegacy‚Äù flow by requiring an **API key** at the edge and only enabling the downgrade for authorized requests (`HTTPRoute` + `AuthPolicy`).

### Quick Run (Kuadrant)

```bash
./ossm/uc11-kuadrant-verify.sh east
```

Expected:

- `GET /modern` ‚Üí **502** (no downgrade, legacy TLS handshake fails downstream)
- `GET /legacy` without API key ‚Üí **401**
- `GET /legacy` with API key ‚Üí **200**

## What to verify in Kiali Traces (expected)

### Istio-only variant (expected)

When you test UC11 (Istio-only), you should see two connector operations with different outcomes:

- **Modern/strict path (expected failure)**:
  - **Operation**: `legacy-connector-modern.egress-legacy.svc.cluster.local:8080/*`
  - **Response status**: **503**
- **Compat path (expected success)**:
  - **Operation**: `legacy-connector-compat.egress-legacy.svc.cluster.local:8080/*`
  - **Response status**: **200**

### Kuadrant variant (expected)

From the `legacy-probe` workload, requests to the edge Gateway should show:

- `GET .../modern` ‚Üí **502**
- `GET .../legacy` ‚Üí **200** (authorized) and **401** (unauthorized)

Seeing **502/503/401** in Kiali is expected here because the demo intentionally exercises the ‚Äúbad‚Äù paths as controlled failures.
