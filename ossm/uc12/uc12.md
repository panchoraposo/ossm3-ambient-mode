# UC12: Blue/Green Deployment with Gateway API

## Objective

Demonstrate weight-based and header-based traffic routing between two versions of a service using Kubernetes Gateway API (HTTPRoute), processed by the mesh ingress gateway (Envoy/istiod). No sidecars, no VirtualService, no external load balancers, zero pod restarts.

## Architecture

```
                     HTTPRoute (weights / headers)
                            ┌──────┐
curl /reviews/0 ─▶ Route ──┤ GW   ├──▶ reviews-green (v1) — no stars
                            │Envoy │
                            │      ├──▶ reviews-blue  (v3) — red stars
                            └──────┘
                    Processed by istiod (mesh control plane)
```

- **Green (v1)**: reviews without ratings (no stars)
- **Blue (v3)**: reviews with red stars
- Both versions already deployed as `reviews-v1` and `reviews-v3` in `bookinfo`
- Traffic routing happens at the **mesh ingress gateway** via Gateway API HTTPRoute
- The HTTPRoute is a mesh resource processed by istiod and enforced by the gateway's Envoy proxy
- Verification via `curl` to `/reviews/0` (routed through the gateway)
- No VirtualService — pure Gateway API (Stable in OSSM 3.2)

## Prerequisites

- EAST cluster running with bookinfo deployed
- **IMPORTANT** `generate-traffic.sh` **stopped** (its traffic to `/productpage` bypasses gateway routing and pollutes the Kiali graph with the original `reviews` service)
- Kiali open (OSSMC via ACM console):
  https://console-openshift-console.apps.cluster-72nh2.dynamic.redhatworkshops.io/ossmconsole/graph

## Quick Run

```bash
./ossm/uc12-verify.sh
```

## T1: The Setup — Two Versions, One Route

Create versioned Services that select specific review versions:

```bash
oc --context east apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  labels:
    app: reviews
  name: reviews-green
  namespace: bookinfo
spec:
  ports:
  - port: 9080
    name: http
  selector:
    app: reviews
    version: v1
---
apiVersion: v1
kind: Service
metadata:
  labels:
    app: reviews
  name: reviews-blue
  namespace: bookinfo
spec:
  ports:
  - port: 9080
    name: http
  selector:
    app: reviews
    version: v3
EOF
```

Update the HTTPRoute with weight-based routing (100% green):

```bash
oc --context east -n bookinfo patch httproute bookinfo --type=json -p '[
  {"op":"replace","path":"/spec/rules/2/backendRefs",
   "value":[
     {"name":"reviews-green","port":9080,"weight":100},
     {"name":"reviews-blue","port":9080,"weight":0}
   ]}
]'
```

Verify: All traffic come into pods in `reviews-v1` the "green"


```bash
curl http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/reviews/0
# → always {... "podname": "reviews-v1-xxxx" ...} (green, no stars)
```

## T2: Step-by-Step Demo Execution

### A. Initial State: 100% Green

**What happens**: The HTTPRoute sends all `/reviews` traffic to `reviews-green` (v1, no stars).

**Show**: The HTTPRoute YAML with `weight: 100` on green, `weight: 0` on blue.

```bash
oc --context east get httproute bookinfo -n bookinfo -o yaml | grep -A5 "reviews"
```

**Kiali**: All traffic flows from the gateway to `reviews-green`. No traffic to `reviews-blue`.

### B. The "Canary" / Smoke Test (90/10)

**What happens**: 10% of traffic starts reaching the new version (blue/v3) while 90% stays on green/v1.

**Action**: Single patch command — instant, no pod restart:

```bash
oc --context east -n bookinfo patch httproute bookinfo --type=json -p '[
  {"op":"replace","path":"/spec/rules/2/backendRefs/0/weight","value":90},
  {"op":"replace","path":"/spec/rules/2/backendRefs/1/weight","value":10}
]'
```

**Verify**:

```bash
for i in $(seq 1 10); do
  curl -s http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/reviews/0 \
    | python3 -c "import sys,json; print(json.load(sys.stdin).get('podname','?'))"
done
```

**Kiali**: The traffic line splits — a thick line to `reviews-green` and a thin line to `reviews-blue`.

**Key point**: This is the moment to compare response times and error codes between versions. If v3 has issues, roll back by setting weights back to 100/0 — instant, no deployment needed.

### C. The "Blue" Promotion (The Switch)

**What happens**: All traffic flips to blue (v3, red stars). Green receives zero traffic.

**Action**: Another single patch — instant traffic shift:

```bash
oc --context east -n bookinfo patch httproute bookinfo --type=json -p '[
  {"op":"replace","path":"/spec/rules/2/backendRefs/0/weight","value":0},
  {"op":"replace","path":"/spec/rules/2/backendRefs/1/weight","value":100}
]'
```

**Verify**:

```bash
curl http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/reviews/0
# → always reviews-v3 (instant switch, no DNS delay)
```

**Kiali**: 100% of traffic goes to `reviews-blue`. The `reviews-green` node goes idle.

### D. Header-Based Routing (A/B Testing)

**What happens**: Requests with `x-beta-user: true` go to Blue; all other requests go to Green.

**Action**: Apply header-based match in the HTTPRoute:

```bash
oc --context east apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: bookinfo
  namespace: bookinfo
spec:
  parentRefs:
  - name: bookinfo-gateway
  rules:
  - matches:
    - path: { type: PathPrefix, value: /productpage }
    - path: { type: PathPrefix, value: /static }
    - path: { type: PathPrefix, value: /login }
    - path: { type: PathPrefix, value: /logout }
    - path: { type: PathPrefix, value: /api/v1/products }
    backendRefs:
    - name: productpage
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /details }
    backendRefs:
    - name: details
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /reviews }
      headers:
      - name: x-beta-user
        value: "true"
    backendRefs:
    - name: reviews-blue
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /reviews }
    backendRefs:
    - name: reviews-green
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /ratings }
    backendRefs:
    - name: ratings
      port: 9080
EOF
```

**Verify**:

```bash
# Default → Green
curl http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/reviews/0
# → reviews-v1

# Beta user → Blue
curl -H "x-beta-user: true" http://bookinfo.apps.cluster-64k4b.64k4b.sandbox5146.opentlc.com/reviews/0
# → reviews-v3
```

### E. Cleanup

```bash
# Restore original HTTPRoute
oc --context east apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: bookinfo
  namespace: bookinfo
spec:
  parentRefs:
  - name: bookinfo-gateway
  rules:
  - matches:
    - path: { type: PathPrefix, value: /productpage }
    - path: { type: PathPrefix, value: /static }
    - path: { type: PathPrefix, value: /login }
    - path: { type: PathPrefix, value: /logout }
    - path: { type: PathPrefix, value: /api/v1/products }
    backendRefs:
    - name: productpage
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /details }
    backendRefs:
    - name: details
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /reviews }
    backendRefs:
    - name: reviews
      port: 9080
  - matches:
    - path: { type: PathPrefix, value: /ratings }
    backendRefs:
    - name: ratings
      port: 9080
EOF

# Delete versioned services
oc --context east delete svc reviews-green reviews-blue -n bookinfo
```

## Expected Results

| Phase | curl /reviews/0 result |
|-------|------------------------|
| 100% Green | Always reviews-v1 (no stars) |
| 90/10 Canary | ~90% v1, ~10% v3 |
| 100% Blue | Always reviews-v3 (red stars) |
| Header: no header | reviews-v1 (green) |
| Header: x-beta-user: true | reviews-v3 (blue) |
| After cleanup | Round-robin v1/v2/v3 (original behavior) |

## Timing Reference

| Transition | Command | Time to take effect |
|-----------|---------|:------------------:|
| 100% Green → 90/10 | `patch httproute` | < 2 seconds |
| 90/10 → 0/100 | `patch httproute` | < 2 seconds |
| 0/100 → 100/0 (rollback) | `patch httproute` | < 2 seconds |

All transitions are a single `oc patch` command. No deployments, no pod restarts, no DNS changes.

## What is Service Mesh here

| Component | Role | Mesh feature? |
|-----------|------|:------------:|
| HTTPRoute | Defines weight/header routing rules | Yes — Gateway API |
| bookinfo-gateway | Envoy proxy that enforces routing | Yes — mesh ingress gateway |
| istiod | Processes HTTPRoute, configures Envoy | Yes — mesh control plane |
| Kiali | Visualizes traffic split in real-time | Yes — mesh observability |
| reviews-green/blue Services | Kubernetes Services selecting v1/v3 | Kubernetes (support) |

All traffic routing decisions are made by the mesh (istiod + Envoy gateway). The versioned Services are standard Kubernetes resources used as routing targets.

## Key Differentiators

| Feature | How demonstrated | Benefit |
|---------|-----------------|---------|
| No sidecar restart | Pod restarts = 0 throughout | Zero downtime |
| Weight-based routing | HTTPRoute weights (100/0 → 90/10 → 0/100) | Gradual, data-driven migration |
| Header-based routing | `x-beta-user: true` header routes to blue | A/B testing for specific users/clients |
| Instant switch | Weight change = instant traffic shift | No DNS propagation, no F5 |
| Integrated observability | Kiali Graph shows traffic split | Visual governance |
| Gateway API | No VirtualService needed | Future-proof, standard Kubernetes API |

## Key Takeaway

Blue/Green and Canary deployments are achieved through Kubernetes Gateway API (HTTPRoute) at the mesh ingress gateway. The routing is processed by istiod and enforced by the gateway's Envoy proxy — this is 100% Service Mesh traffic management. Traffic shifting is instant (no DNS delays, no external LB reconfiguration), requires zero pod restarts, and is fully observable in Kiali.
